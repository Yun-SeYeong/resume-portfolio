1-1. 점검 실시간 로그

폴링 기반 로그 조회를 Reactor Sinks + WebSocket 실시간 스트리밍으로 전환

문제

진행 중 로그를 실시간으로 보여주는 구조가 없어 클라이언트가 DB를 주기적으로 폴링했습니다. 그 결과 폴링 주기만큼 반영이 늦고, 반복 조회로 DB 부하가 늘며, 중간 접속 시 누적 로그를 한꺼번에 조회하는 지연이 있었습니다.

해결

RabbitMQ로 도착하는 진행 로그를 메모리에 보관하면서 WebSocket으로 실시간 전달하는 구조로 전환했습니다.

  • 메모리 보관 — RabbitMQ 이벤트를 ConcurrentHashMap<examinationStepId, TreeSet<Event>>에 저장해 중간 접속 클라이언트에게 기존 로그를 전달합니다.
  • 실시간 발행 — 이벤트 도착 시점에 Sinks.many().multicast()로 발행해 연결된 모든 WebSocket 클라이언트에 즉시 전달합니다.
  • 중간 접속 처리 — 접속 시 메모리의 기존 로그를 먼저 전송한 뒤 Flux 구독을 시작합니다. 언제 접속하든 로그가 끊기지 않습니다.
  • 연결 종료 — 세션별 Disposable을 관리하고 종료 시 dispose()로 구독을 해제합니다.
설계 트레이드오프
  • multicast() vs replay() — 이 Sinks는 로그뿐 아니라 "점검 수정/취소" 같은 일회성 커맨드도 발행합니다. replay()는 중간 접속 시 과거 커맨드까지 재전송하므로, multicast()를 택하고 과거 로그만 eventMap에서 선별 전달했습니다.
  • Redis Pub/Sub 대신 인메모리 Sinks — 점검 이벤트는 실행 중에만 유효하고 끝나면 사라지는 짧은 수명의 데이터입니다. 외부 시스템 도입은 인프라 복잡도만 키우므로 단일 인스턴스 내 Reactor Sinks가 적합했습니다.
  • 메모리 누수 방지 — @EventListener로 ExaminationFinishedEvent를 감지해 5초 후 데이터를 삭제합니다. 5초 지연은 완료 직후 접속한 클라이언트가 마지막 상태를 받도록 한 장치입니다.

1-2. 패키지 관리 시스템

대용량 패키지 파일 CRUD + K8s 환경의 Pod 간 파일 공유 인프라 설계

문제

네트워크 장비에 배포할 소프트웨어 패키지(이미지·패치·릴리즈 노트·설정 파일)를 업로드·관리해야 했습니다. 파일이 수 GB로 크고 여러 첨부 타입을 함께 다루며, 웹에서 바로 다운로드할 수 있어야 했습니다.

  • API — 패키지 CRUD와 함께 5가지 파일 타입(PKG_IMAGE, PKG_PATCH, RELEASE_NOTE, MODIFIED_CONFIG, PKG_VERIFICATION_RESPONSE)을 Multipart로 업로드·개별 삭제하는 REST API를 구현했습니다.

  • 인프라 — as-core가 업로드하면 같은 PV를 마운트한 Nginx Pod가 웹에서 바로 서빙합니다.
  • Helm Chart — Nginx StatefulSet + PVC(100Gi, nfs-client) + ConfigMap + ClusterIP Service로 환경별 배포를 관리합니다.
설계 트레이드오프
  • Pod 간 파일 공유 — K8s Pod은 파일시스템이 격리되므로 NFS 기반 PV를 ReadWriteMany로 생성해 여러 Pod이 동시에 읽기/쓰기하도록 했습니다.
  • 서빙 역할 분리 — 수 GB 파일을 Spring Boot(Tomcat)가 직접 서빙하면 다운로드 동안 스레드를 점유해 다른 API에 영향을 줍니다. 정적 파일에 최적화된 Nginx에 서빙을 맡기고 as-core는 비즈니스 로직에 집중하도록 분리했습니다.

2. NodeServer 구축여러 서비스가 공유하는 장비 정보를 통합 제공하는 서비스 신규 구축

2-1. 장비 인벤토리 통합 조회 API

10여 개 조건을 조합하는 QueryDSL 동적 검색 + N+1 없는 Fetch Join

문제

장비는 벤더·모델·OS 버전·네트워크 계층·설치 위치 등 다양한 속성을 가지며, 서비스마다 서로 다른 조건으로 필터링해 조회해야 했습니다.

  • 동적 필터링 — 10개 이상의 조건(주소·시스템 이름·벤더·모델·OS 버전·계층·등록일 범위 등)을 null-safe하게 조합하는 동적 WHERE절로, 하나의 쿼리가 모든 조합을 처리합니다.
  • Fetch Join — 연관 엔티티(벤더·모델·네트워크 타입·MTSO)를 LEFT JOIN FETCH로 N+1을 방지하고, 페이지네이션은 별도 count 쿼리로 분리했습니다.
  • 필터 값 API — UI 드롭다운 값을 현재 데이터 기반으로 동적 생성해 벤더→모델→OS 버전 계층으로 반환합니다.
설계 트레이드오프
  • Fetch Join과 페이지네이션 분리 — 둘을 함께 쓰면 Hibernate가 메모리에서 페이징하는 문제가 있습니다. 데이터 조회 쿼리와 count 쿼리를 분리해 조회에만 Fetch Join을 적용했습니다.

2-2. 점검/작업 시 장비 리스크 분석

토폴로지 기반으로 동시 작업 위험 장비 그룹을 O(n)으로 식별

문제

같은 Ring(환형 토폴로지)이나 MTSO(교환국)를 공유하는 장비를 동시에 점검하면 Ring 전체가 다운될 수 있습니다. 예약 시 동시에 작업하면 안 되는 장비 그룹을 사전에 식별해야 했습니다.

  • 리스크 그룹 분류 — 장비 IP 목록을 토폴로지 기반으로 그룹핑합니다. 추가 시 Ring ID·MTSO가 기존 그룹과 겹치면 새 그룹으로 분리해, 동시에 작업해도 안전한 장비가 같은 그룹에 묶입니다.
  • 중복 리스크 감지 — 같은 Ring에 여러 장비가 있거나 같은 MTSO에 집중된 경우를 Ring 중복·MTSO 중복으로 나눠 반환합니다.
  • 네트워크 타입별 기준 — Backbone은 MTSO 다양성을, Backhaul은 Ring 동질성과 시설 코드 고유성을 기준으로 분석합니다.
설계 트레이드오프
  • 중복 판단의 시간복잡도 — 장비 쌍마다 비교하면 O(n²)입니다. HashSet에 Ring ID를 누적하며 추가 전후 크기 변화를 비교(O(1))하는 방식으로 전체를 O(n)으로 개선했습니다. 예: 기존 3개에 Ring 2개짜리 장비 추가 후 크기가 5면 겹침 없음, 4면 하나 겹침.

3. CLI Commander API 구축여러 장비에 동시에 CLI 명령을 보내고 결과·이력을 관리하는 서비스 신규 구축

3-1. 비동기 명령 실행 파이프라인

HTTP 동기 요청 + RabbitMQ 비동기 결과 수신으로 블로킹 제거

문제

여러 장비에 동시에 명령을 보내면 응답 시간이 제각각이고 수 분 이상 걸릴 수 있습니다. 동기로 처리하면 API 서버가 그동안 블로킹되므로, 요청과 결과 수신을 분리해야 했습니다.

  • 명령 제출 — NodeServer에서 장비 정보를 조회한 뒤 워커에 HTTP로 작업을 요청하며, 이때 UUID와 결과 수신용 큐 이름을 함께 전달합니다.
  • 결과 수신 — 워커가 결과를 큐에 발행하면 CommandEvent(명령 단위)와 HostEvent(장비별)를 구독해 수신합니다.
  • 이력 적재 — 이벤트를 UUID로 매칭해 CommandHistory에 누적하며, 최종 상태는 가장 마지막 이벤트의 status로 판단합니다.
설계 트레이드오프
  • 요청/결과의 분리 — 요청은 UUID 생성 후 즉시 응답해야 하므로 HTTP 동기로, 결과는 실행 시간에 따라 비동기로 도착하므로 RabbitMQ로 수신합니다. 요청 시 resultQueueName을 전달해 워커가 어느 큐로 보낼지 알게 했습니다.
  • 이벤트 기반 상태 추적 — 상태를 분리 컬럼으로 두지 않고 이벤트를 쌓아 마지막 이벤트(max count)의 status를 최종 상태로 판단해, 중간 상태 이력도 자연스럽게 보존했습니다.

3-2. 장비 계정 검증

Reactor Sinks 이벤트 버스 + 10초 타임아웃으로 무한 대기 방지

문제

장비 계정의 유효성을 실제 접속으로 확인해야 하는데, 응답이 느리거나 장비 상태에 따라 영원히 오지 않을 수도 있었습니다.

  • 이벤트 버스 — 결과가 RabbitMQ로 도착하면 Sinks.many().multicast()로 발행하고, 요청자는 UUID로 필터링된 Flux를 구독해 자신의 결과만 받습니다.
  • 완료 감지 — Flux.takeUntil로 status가 finished인 이벤트가 도착하면 수집을 종료합니다.
  • 타임아웃 — 10초 내 응답이 없으면 onErrorResume으로 timeout 상태 응답을 생성해 항상 응답합니다.
설계 트레이드오프
  • 동시 요청과 UUID 매칭 — 여러 운영자의 결과가 하나의 큐로 도착합니다. Sinks 이벤트 버스 + UUID 필터링으로 요청자별 독립 Flux 스트림을 제공했습니다.

3-3. 명령 이력 관리

키워드·상태·기간을 조합하는 동적 검색, 상태는 서브쿼리로 정확히 필터링

  • 동적 검색 — 키워드(이름·사용자·호스트네임)·상태·날짜 범위·사용자 필터를 QueryDSL로 조합합니다.
  • 장비별 결과 — 하나의 명령에 포함된 장비별 실행 상태와 stdout/stderr를 개별 확인합니다.
설계 트레이드오프
  • 상태 필터링 서브쿼리 — "성공한 이력만"을 단순히 successful 이벤트 유무로 보면, 중간에 successful이었다가 failed로 끝난 이력도 걸려옵니다. 서브쿼리로 마지막 이벤트(max count)의 status를 기준으로 필터링해 정확한 최종 상태로 검색했습니다.

4. 기간망 선번장 관리 API광 네트워크 경로(선번장)의 자동/수동 데이터 비교와 양방향 동적 검색

4-1. 자동/수동 선번장 데이터 비교

PK가 다른 두 데이터셋을 Endpoint 조합 정규화로 매칭

문제

선번장은 EMS 자동 수집(Linkbook)과 운영자 수동 등록(Passivity)이 별도로 존재합니다. PK 구조가 달라 DB 조인이 불가능하지만, 벤더별로 두 데이터를 대조해 누락·불일치를 확인해야 했습니다.

  • 이중 엔티티 — 자동(CmUiLinkbook)과 수동(CmUiLinkbookPassivity)을 별도 엔티티로 관리하고, 각 선번장은 시작/끝 구간별 12개 Embedded 객체(StVs·StTc·StIo·StLk·StMux·StSp·Ed…)로 광 경로를 표현합니다.
  • 조합 기반 매칭 — endpoint 조합을 비트 연산으로 모든 순서로 생성한 뒤 TreeSet으로 정규화하고, 이를 HashMap 키로 사용해 두 데이터셋을 매칭합니다. 결과를 자동에만/수동에만/양쪽 일치로 분류합니다.
  • 미관리 분리 — 제외 대상 서비스를 사유와 함께 기록하고 비교 시 관리/미관리를 분리해 처리합니다.
설계 트레이드오프
  • PK가 다른 두 데이터셋 매칭 — 같은 물리 경로라도 자동은 A→B, 수동은 B→A로 입력될 수 있습니다. 비트 연산으로 모든 순서 조합을 생성하고 TreeSet의 자연 정렬로 정규화해, 기록 순서와 무관하게 동일 경로를 식별했습니다.

4-2. 다양한 필터 조건의 동적 검색

방향 있는 광 경로를 양방향 QueryDSL Predicate로 한 번에 검색

문제

선번장은 70개 이상의 필드를 가지며, 5개 EndType(IO·LK·MUX·TC·VS)별 시작/끝 구간의 station·node·shelf·slot·port를 조합해 검색해야 합니다. 광 경로는 방향이 있어 시작·끝을 모를 때도 검색돼야 했습니다.

  • 양방향 Predicate — "A→B" 검색 시 "B→A"도 함께 검색하도록 ST↔ED를 뒤집어 적용해, 한 번의 쿼리로 양방향 결과를 가져옵니다.
  • EndType별 매핑 — 5개 EndType에 따라 다른 Embedded 필드 경로를 매핑하며, IO 타입은 양쪽 노드 쌍(stIoStNodeName·stIoEdNodeName)을 별도 처리합니다.
  • 필터 옵션 동적 생성 — 벤더·서비스타입·회선종류·채널·파장·중계경로 등을 현재 데이터에서 추출해 실제 존재하는 값만 표시합니다.
Project 02

국제망 관리 시스템

11개 해외 통신사 벤더의 국제 회선 생애주기를 관리하는 시스템입니다. 국내·해외·해저케이블 구간을 거치는 회선의 등록, 경로 설정, 장애 관리, 장비 인벤토리를 통합 처리하며, 65개 이상의 엔티티로 복잡한 도메인을 모델링했습니다.

Spring BootJPA @Inheritance QueryDSLPostgreSQL Mail Parsing@Scheduled

1. 6가지 경로 타입의 통합 도메인 모델링@Inheritance(JOINED)로 6개 경로 타입을 통합하고 다형적으로 조회

문제

국제 회선 경로는 국내 XC, 국내 Backhaul, 해외 XC, 해외 Backhaul, Trunk, 해저케이블 등 6가지 타입이 있습니다. 타입마다 고유 필드가 다르지만 하나의 회선에서 순서대로 연결되므로 통합 관리가 필요했습니다.

  • @Inheritance(JOINED) — 추상 Path 아래 DomesticXCPath·OverseaBackhaulPath·SubmarineCablePath 등 6개 하위 클래스로 분리하고, 공통 필드는 부모 테이블, 고유 필드는 자식 테이블에 저장합니다.
  • 다형적 조회 — 추상 메서드 getCids()·getCidAndProvider()로 각 타입이 자신의 식별자를 반환해, 상위 코드가 타입을 몰라도 통합 조회됩니다.
  • 일관된 패턴 — Trunk도 추상 TrunkPathInfo 아래 6개 하위 타입으로 동일한 모델을 유지했습니다.
설계 트레이드오프
  • JOINED vs SINGLE_TABLE vs TABLE_PER_CLASS — SINGLE_TABLE은 6타입의 고유 필드가 한 테이블에 모여 NULL 컬럼이 과도하고, TABLE_PER_CLASS는 통합 조회 시 UNION ALL로 성능·복잡도가 올라갑니다. JOINED로 정규화 스키마를 유지하면서 @DiscriminatorColumn으로 다형적 조회가 가능하도록 했습니다.

2. 장애 유형의 통합 및 추적자사·벤더·해저케이블 장애를 양방향으로 연결하고 상태 머신으로 추적

문제

자사 장애와 해외 벤더 장애가 서로 영향을 주고 이력을 양방향으로 추적해야 합니다. 해저케이블 장애는 물리 구간 단위로 별도 관리되며, 각 유형마다 상태·이력을 독립적으로 추적해야 했습니다.

  • 자사 장애 (SkbFailureTicket) — "날짜+시퀀스"로 ID를 자동 생성하고 SkbFailureHistory로 이력을 추적하며, ProviderFailureTicket과 양방향 참조로 연결합니다.
  • 벤더 장애 (ProviderFailureTicket) — 벤더 메일 파싱으로 자동 생성되고, CID로 회선을 식별하며 IplMail로 원본 메일을 추적합니다.
  • 해저케이블 장애 (SubmarineCableFailure) — Cable → Segment 계층에서 장애 구간을 특정하고 유형·경로·장애/복구 시간을 기록합니다.
  • 상태 머신 — 접수 → 확인 → 복구 → 완료 4단계로 관리합니다.
설계 트레이드오프
  • 자사–벤더 양방향 연결 — 하나의 자사 장애에 여러 벤더 장애가 연결될 수 있고, 메일에서 생성된 장애를 기존 자사 장애에 매칭해야 합니다. 두 티켓을 양방향 OneToMany/ManyToOne으로 연결하고, 파싱 시 CID로 관련 자사 티켓을 자동 검색해 연결했습니다.

3. 11개 벤더 메일 자동 파싱 파이프라인형식이 모두 다른 11개 벤더 메일을 인터페이스 추상화로 자동 파싱

문제

11개 벤더가 장애/유지보수 알림을 이메일로 보내는데 형식이 모두 다릅니다. 날짜 포맷·회선 ID 표기·본문 구조·타임존이 제각각이라 수동 입력은 비효율적이고 누락 위험이 컸습니다.

  • MailParser 인터페이스 — getCID·getTID·getStartWorkTime·getEndWorkTime·getWorkDesc·getEffect 등 공통 계약을 정의하고 벤더별 11개 구현체를 만들었습니다.
  • 벤더 자동 식별 — checkFrom()으로 발신자 주소 기반 식별하고, 파싱 실패 시 ErrorMailParser로 원본을 보존해 누락을 방지합니다.
  • 타임존 정규화 — IST·CST·AEST 등 벤더별 타임존을 ZonedDateTime으로 파싱한 뒤 KST로 변환하는 공통 유틸을 만들었습니다.
  • @Scheduled 수집 — 주기적으로 메일을 가져와 자동 파싱하고, 결과(CID·TID·시간·영향도)를 Keyword 엔티티로 저장해 장애 티켓과 연결합니다.
설계 트레이드오프
  • 형식 차이 대응 — CID가 제목·본문·첨부(Excel)에 있는 벤더가 제각각이고 날짜도 DD/MM/YYYY·YYYY-MM-DD 등 다양합니다. MailParser 인터페이스로 추상화하고 정규식·파싱 로직을 구현체에 맡겨, 새 벤더는 인터페이스만 구현하면 되도록 했습니다.
Project 03

L3 네트워크 장비 관리 시스템

통신사 L3 스위치(Huawei/Arista)의 구성 조회·설정 변경·텔레메트리 수집을 위한 분산 처리 시스템입니다. 4개 프로토콜(CLI·NETCONF·eAPI·gNMI)을 통합 인터페이스로 추상화하고, RabbitMQ 기반 비동기 파이프라인으로 수백 대 장비를 병렬 처리합니다. Gradle 멀티모듈 11개로 API Server·Collector·Batch Server 등을 독립 배포합니다.

Spring BootgRPC · Protobuf RabbitMQNETCONF · eAPI · gNMI PostgreSQLGradle 멀티모듈

1. 멀티벤더·멀티프로토콜 추상화"무엇을"과 "어떻게"를 분리한 이중 Strategy로 벤더·프로토콜을 독립 확장

문제

Huawei 3종(V2R5·V2R19·V8R12)과 Arista 3종(4.22·4.25·4.26)을 관리해야 합니다. 벤더마다 지원 프로토콜이 다르고(Huawei: CLI/NETCONF, Arista: CLI/eAPI/gNMI), 같은 기능도 명령어 형식(XML·JSON-RPC·protobuf·텍스트)이 완전히 다릅니다.

  • CmdService (무엇을) — 벤더+버전별 6개 구현체가 명령 생성(getXxxCmd)과 응답 파싱(getXxxParser)을 담당합니다. Spring의 Map<String, CmdService> 자동 주입으로 벤더/버전에서 Bean 이름을 조합해 런타임에 선택합니다.
  • CollectService (어떻게) — 프로토콜별 4개 구현체(CollectCli·CollectNetConf·CollecteAPI·CollectgNMI). CmdService가 명령마다 지정한 프로토콜에 따라 자동 선택됩니다.
  • 벤더별 전처리 — Huawei는 screen-length 0 temporary, Arista는 enable\nterminal length 0으로 CLI 출력 페이징을 비활성화하는 등 전처리를 분리했습니다.
설계 트레이드오프
  • Strategy 이중 적용 — CmdService는 벤더+버전, CollectService는 프로토콜 기준으로 전략을 선택해 두 변화를 독립 처리합니다. 실제로 Arista 4.26 추가 시 CmdArista_4_26_1F 클래스 하나에 @Service만 붙여 완료되었습니다.

2. RabbitMQ 기반 비동기 Job 파이프라인Topic Exchange 동적 바인딩으로 수백 대 병렬 처리 + Collector Scale In/Out

문제

수백 대 장비에 동시에 명령을 보내고 결과를 수집해야 합니다. 동기 처리하면 장비 응답 대기로 API 서버가 블로킹되고, Collector의 Scale In/Out으로 처리량에 유연하게 대처해야 했습니다.

  • 동기/비동기 전환 — callback URL이 있으면 비동기, 없으면 동기입니다. 동기 모드는 RxJava PublishSubject로 UUID 기반 Complete Message를 수신하며 3분 타임아웃으로 무한 대기를 방지합니다.
  • Topic Exchange + 동적 바인딩 — 장비 IP를 Routing Key로 사용해 같은 장비 명령이 같은 Queue로 전달됩니다. 바인딩 없는 IP가 오면 바인딩이 가장 적은 Queue에 자동 생성하고, 완료 후 불필요한 바인딩을 제거합니다.
  • Collector 워커 — 20개 Queue에 150개 Consumer가 분산 배치되어, JobMessage를 받아 CollectService로 실행하고 CompleteMessage를 Complete Exchange로 발행합니다.
설계 트레이드오프
  • Exchange 전략 — Consistent Hash는 분포 편향, Modulus Hash는 Collector 추가/제거 시 재배정이 안 됩니다. Topic Exchange로 바인딩을 직접 제어하고, Scale In/Out 시 평균 기준으로 재분배하는 로직을 구현했습니다.
  • Scale In/Out 대응 — Collector는 시작 시 consumer-count만큼 Queue를 동적 생성하고, RabbitMQ Management HTTP API로 바인딩 상태를 조회합니다. 바인딩 없는 IP가 도착하면 평균보다 낮은 Queue에 생성하고, 완료 후 평균+1보다 많은 Queue의 바인딩을 제거해 균형을 유지하며, 이 과정을 synchronized로 동시성 보장합니다.
  • 추가/제거 시나리오 — Collector 추가 시 빈 Queue가 생기며 전체 평균이 낮아져 기존 바인딩이 새 Queue로 재배정됩니다. 제거 시 해당 Queue가 삭제되고 미처리 메시지는 timeout 처리, 바인딩은 다음 요청에 다른 Queue로 재배정됩니다.
